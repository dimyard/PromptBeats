# Общая библиотека треков — дизайн

Дата: 2026-07-11
Ветка/worktree: `feat/shared-library`
Затрагивает: компонент A (фронт), компонент B (бэк), Контракт (новый Контракт 5).

## Цель

Дать пользователю сохранять текущий трек в **общую** (серверную) библиотеку, просматривать
библиотеку в выезжающей панели и загружать/воспроизводить трек из неё. Библиотека общая:
все, кто открывают приложение с одним бэкендом, видят одну коллекцию.

Формат трека для хранения — уже существующий валидный **Song JSON v1** (та же форма, что
`sample-song.json` и файл экспорта). Схема (`song.schema.json`) **не меняется**.

## Не входит в объём (YAGNI)

- Удаление треков из библиотеки.
- Авторство/аккаунты/аутентификация.
- Шаринг по ссылке, пагинация, поиск, теги.
- Изменение формы Song JSON или `/api/compose`.

## Решения из брейншторминга

- **Где живёт:** на бэкенде, реально общая (файловое хранилище).
- **Операции:** сохранить текущий; просмотр + загрузка/play; перезаписать/обновить. Удаления нет.
- **Дедуп при сохранении:** проверять и по названию, и по самому JSON. Если дубль — сообщить
  «такой трек уже есть, вот он».
- **Подпись:** только название трека (из `song.title`, редактируемо при сохранении).
- **UI:** выезжающая панель (drawer) справа.

Разделение обязанностей:
- `backend/src/library.js` — вся логика: чтение/запись файла, дедуп, апдейт. Тестируемо в Node

- `backend/src/server.js` — только HTTP-обвязка: разбор запроса, вызов library.js, конверт ответа/ошибки.
- `frontend/src/library-api.js` — тонкий fetch-клиент (в стиле `api.js`).
- `frontend/src/App.jsx` — состояние drawer, обработчики save/load, компонент `LibraryDrawer`.

## Контракт 5 — Общая библиотека (A ↔ B)

Добавляется в `CONTRACTS.md`. `song.schema.json` и `/api/compose` не меняются.
Конверт ошибок тот же, что у `/api/compose`: `{ error: { code, message } }`.

### Запись в хранилище (внутренняя форма)
```jsonc
{
  "id": "lib_<уникальный>",          // серверный id
  "title": "lofi sketch",             // из song.title (или переданного title)
  "createdAt": "2026-07-11T10:00:00.000Z",
  "updatedAt": "2026-07-11T10:00:00.000Z",
  "song": { /* полный валидный Song JSON v1 */ }
}
```

### `GET /api/library` — список (лёгкий)
Response `200`:
```jsonc
{
  "tracks": [
    {
      "id": "lib_abc",
      "title": "lofi sketch",
      "createdAt": "…", "updatedAt": "…",
      "summary": { "bpm": 75, "bars": 8, "key": "A minor", "tracks": 3 }
    }
  ]
}
```
Без поля `song` — чтобы список был лёгким. Сортировка: свежие `updatedAt` сверху.

### `GET /api/library/:id` — один трек (полный)
Response `200`:
```jsonc
{ "track": { "id": "lib_abc", "title": "…", "createdAt": "…", "updatedAt": "…", "song": { /* Song JSON */ } } }
```
Нет такого id → `404 { "error": { "code": "not_found", "message": "…" } }`.

### `POST /api/library` — сохранить
Request body:
```jsonc
{
  "song": { /* полный Song JSON */ },   // обязательно, валидируется как в /api/compose
  "title": "необязательно",              // если нет — берётся song.title
  "overwrite": false                     // подтверждение перезаписи при конфликте имён
}
```
Валидация входа зеркалит `/api/compose` (объект с массивом `tracks`, лимит размера
`MAX_SONG_CHARS`). Иначе `400 bad_request`.

Перед дедупом и сохранением эффективное имя `effectiveTitle = title ?? song.title` записывается
в `song.title`. Поэтому каноническое сравнение контента естественно учитывает выбранное имя,
а сохранённая запись самосогласована (её `title` совпадает с `song.title`).

Response `200` — всегда с полем `status` (это не ошибки, а штатные ветки):

| status | когда | тело |
|--------|-------|------|
| `created` | нового такого нет | `{ status:"created", track: <meta> }` |
| `duplicate` | идентичный JSON уже есть | `{ status:"duplicate", track: <meta существующего> }` (новая запись НЕ создаётся) |
| `title_conflict` | то же имя, другой контент, `overwrite` не задан | `{ status:"title_conflict", track: <meta существующего> }` (НЕ сохраняем) |
| `updated` | то же имя + `overwrite:true` | `{ status:"updated", track: <meta> }` |

`<meta>` — та же форма, что элемент списка (id, title, createdAt, updatedAt, summary).

### Дедуп (canonical JSON)
Идентичность контента = равенство **канонической** сериализации всего `song`
(рекурсивная сортировка ключей объектов; массивы — как есть). Это устойчиво к порядку полей.
Идентичность включает `title` (переименованный трек с тем же битом — НЕ дубль по контенту,
но упрётся в `title_conflict`, если имя совпало; иначе создаётся как новый).

Приоритет проверок на сохранении:
1. Каноническое равенство всего `song` с любой записью → `duplicate`.
2. Иначе совпадение `title` с записью → `updated` (при `overwrite:true`) или `title_conflict`.
3. Иначе → `created`.

### Хранилище и надёжность
- Файл `backend/data/library.json` — массив записей. Каталог `backend/data/` в `.gitignore`
  (runtime-состояние, не коммитим).
- Запись атомарно: во временный файл + `rename`. Обновления сериализованы (простая очередь/мьютекс
  в процессе), чтобы одновременные POST не затирали друг друга.
- Нет файла / битый JSON при чтении → трактуем как пустую библиотеку (лог в консоль), не роняем сервер.
- `id` генерится на сервере (timestamp + случайный суффикс).

## Фронтенд (компонент A)

### `frontend/src/library-api.js` (новый, в стиле `api.js`)
```ts
listLibrary(): Promise<{ tracks: Meta[] }>
getLibraryTrack(id): Promise<{ track: { …, song } }>
saveToLibrary({ song, title?, overwrite? }): Promise<{ status, track }>
```
Ошибки бросают `Error(message)`, как `compose()`.

### `LibraryDrawer` (новый компонент в `App.jsx`)
Выезжает справа поверх студии (студия остаётся видимой), в едином визуальном стиле с
инспектором/import-overlay. Открывается кнопкой **«Библиотека»** в транспорте (верх студии).

Содержимое:
- **Сверху — «Сохранить текущий»:** поле имени (префилл из `song.title`) + кнопка «Сохранить».
  Кнопка disabled без загруженного song.
  - `created` → тост «Сохранено в библиотеку», обновить список.
  - `updated` → тост «Трек обновлён», обновить список.
  - `duplicate` → инлайн-сообщение «Такой трек уже есть: „X"» + подсветка/скролл к нему в списке.
  - `title_conflict` → инлайн-подтверждение «Трек „X" уже есть. Перезаписать?» [Перезаписать][Отмена];
    «Перезаписать» повторяет POST с `overwrite:true`.
- **Ниже — список:** карточка трека = название, мета `bpm · bars · key · N дорожек`, дата;
  действия **«Загрузить»** и **«▶»** (загрузить + сразу play — клик = user-gesture, play разрешён).
  Состояния: загрузка списка (скелет/спиннер), пусто («библиотека пуста»), ошибка/бэкенд недоступен
  (в стиле существующего сообщения «Бэкенд недоступен»).

Загрузка трека переиспользует существующий `loadSong(song, message)` (плеер + чат-сообщение).
После загрузки — тост и (опционально) drawer остаётся открытым для быстрого перебора.

Открытие drawer триггерит `listLibrary()` (свежий список каждый раз, т.к. библиотека общая).

### Состояние в `App.jsx`
`libraryOpen`, `libraryTracks`, `libraryState` (idle|loading|error), `libraryError`,
`saveName`, `pendingConflict` (для инлайн-подтверждения перезаписи).

## Тестирование

- **Бэк — `backend/test/library.test.mjs`** (node --test, в стиле существующих):
  - save нового → `created`, запись в сторе, `id`/timestamps проставлены.
  - повторный save того же song → `duplicate`, стор не растёт.
  - save с тем же title, другой контент, без overwrite → `title_conflict`, не сохранено.
  - тот же с `overwrite:true` → `updated`, контент заменён, `updatedAt` обновлён, число записей то же.
  - `list` → meta без `song`, сортировка по `updatedAt`.
  - `get` по id → полный song; неизвестный id → not_found.
  - canonical-дедуп устойчив к порядку ключей.
  - Стор пишет во временный файл (изоляция теста).
- **Фронт:** канонический сериализатор при необходимости выносится в чистый хелпер и покрывается;
  `library-api.js` — тонкий fetch-враппер (интеграционно проверяется через smoke).
