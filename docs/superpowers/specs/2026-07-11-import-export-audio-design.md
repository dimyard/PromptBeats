# PromptBeats — Импорт/экспорт проекта и сохранение аудио (дизайн)

Дата: 2026-07-11 · Статус: утверждён к реализации · Владельцы: A (UI + song-io), C (player.exportWav)

## 1. Цель

Дать пользователю унести трек из приложения и вернуть его обратно, а также сохранить итоговый микс как аудиофайл:

1. **Экспорт проекта** — скачать текущий Song JSON файлом.
2. **Импорт проекта** — загрузить Song JSON обратно **тремя способами**: вставкой текста, выбором файла, drag-n-drop
   (файл **или** текст). Трек должен переноситься «как есть» и оставаться полностью редактируемым дорожками.
3. **Сохранение аудио** — отрендерить текущий микс в **WAV** и скачать.

Все действия сопровождаются плавными анимациями и понятной индикацией состояния строго по дизайн-системе репозитория.

## 2. Scope

**Делаем:**
- Экспорт текущего `song` в `.json` (файл = валидный Song JSON v1, без обёртки-метаданных).
- Импорт Song JSON: textarea-вставка, file picker, drag-n-drop файла и текста; единый конвейер обработки.
- Клиентская валидация + нормализация импорта (без новых тяжёлых зависимостей).
- `player.exportWav(song)` — офлайн-рендер в WAV (реализация уже объявленного опционального метода Контракта 3).
- Анимации/индикация состояний по дизайн-системе (`docs/design/promptbeats.design-tokens.json`), с уважением к
  `prefers-reduced-motion`.

**Не делаем (YAGNI на этом этапе):**
- Импорт/экспорт **отдельных дорожек** и мердж в текущий Song (только целый проект).
- MP3/OGG и прочие форматы с внешним энкодером (только WAV — нативно, без зависимостей).
- Обёртку-конверт файла проекта (`{app, exportedAt, song}`). Файл = чистый Song JSON, чтобы реимпортировался,
  редактировался и принимался бэком без распаковки.
- Шеринг по ссылке/облако/аккаунты.

## 3. Формат файла проекта

Экспортируемый `.json` — **ровно Song JSON v1**, та же форма, что `sample-song.json` и `song.schema.json`.
`version` **не бампаем** (форма данных не меняется). Следствия:
- файл реимпортируется в приложение и остаётся редактируемым (дорожки, bpm, bars, events);
- тот же файл можно вставить в `POST /api/compose` как `song` для правки через LLM;
- совместимость с уже существующей фикстурой и валидатором бэка «из коробки».

Имя файла: `slug(title || "promptbeats") + ".json"` / `".wav"` (латиница/цифры/дефис, нижний регистр).

## 4. Импорт — три способа, один конвейер

Все три входа сходятся в **единый конвейер** `text → parseSong → normalizeImportedSong → loadSong`:

1. **Вставка текстом** — textarea в диалоге импорта; кнопка «Импортировать» берёт содержимое как `text`.
2. **Файл** — скрытый `<input type="file" accept="application/json,.json">`; выбранный файл читается
   `readFileAsText` → `text`.
3. **Drag-n-drop** — зона приёма ловит `drop`:
   - если в `dataTransfer.files[0]` есть файл → читаем его как `text`;
   - иначе берём `dataTransfer.getData("text")` (перетащенный/вставленный текст) как `text`.
   Глобальный `dragenter` по окну раскрывает overlay импорта — файл можно бросить в любое место приложения.

`parseSong(text)`:
1. `JSON.parse` (ошибка → `{ ok:false, error:"Не удалось разобрать JSON: …" }`);
2. `validateSong(song)` — ключевые инварианты (см. §5); при провале → `{ ok:false, error }` со списком проблем;
3. `normalizeImportedSong(song)` — клампы/дроп (см. §5), собирает `warnings`;
4. `{ ok:true, song, warnings }`.

Успех → `loadSong(song)` (существующий путь: `setSong` + `player.load`) + toast; `warnings` (если есть) —
дополнительным ненавязчивым сообщением, но трек всё равно грузится. Ошибка → сообщение остаётся в диалоге,
состояние приложения не меняется.

## 5. Валидация и нормализация (модуль song-io)

`validateSong(song)` — лёгкая, **без зависимостей**, зеркалит ключевые инварианты `song.schema.json` (держать
в синхроне при правках схемы):
- `version === 1`; `typeof title === "string"` (если есть) ≤ 120;
- `bpm` число 40..220; `bars` целое 1..32;
- `tracks` — массив 1..12; у каждого `id` (`^[a-zA-Z0-9_-]{1,40}$`, уникален), `role` ∈ enum,
  `instrument` ∈ {synth, sampler}, `sound` из каталога, пара **instrument↔sound** валидна (synth→синт, sampler→кит);
- `events[]` — у каждого `step` целое ≥ 0, `note` по паттерну, `dur`/`vel` в допуске (если заданы).

`normalizeImportedSong(song)` — повторяет поведение бэкового `normalizeSong` и защиты плеера, чтобы импорт
«просто работал»:
- дроп событий со `step` вне `0..bars*16-1` (→ warning с количеством);
- кламп `dur`, чтобы `step + dur ≤ bars*16`;
- дедуп повторяющихся `id` дорожек (→ warning);
- кламп `bpm`/`bars`/`gain` в допустимые диапазоны.
Нормализация **идемпотентна**: повторный прогон уже нормализованного Song ничего не меняет.

Инвариант round-trip: `parseSong(serializeSong(song)).song` глубоко равен нормализованному `song`.

## 6. Экспорт проекта (JSON)

`song → serializeSong(song) → Blob("application/json") → downloadBlob(blob, songFilename(song,"json"))`.
`serializeSong` = `JSON.stringify(song, null, 2)` + перевод строки. Кнопка disabled, если `song` не загружен.

## 7. Сохранение аудио (WAV)

`song → player.exportWav(song) → Blob("audio/wav") → downloadBlob(blob, songFilename(song,"wav"))`.

Реализация `exportWav` (зона C, `frontend/src/player/`):
- `Tone.Offline(cb, durationSec)` — офлайн-контекст; внутри `cb` **переиспользуем** `makeSynth`/`makeKit`
  (они биндятся к текущему = офлайновому контексту) и ту же логику расстановки событий, что в `load`;
- длина рендера = длина одного лупа: `durationSec = bars * 4 * 60 / bpm` (= `totalSteps` шестнадцатых).
  Рендерим ровно один луп для бесшовности; release-хвосты синтов на границе лупа обрезаются — **известное
  ограничение MVP**, помечаем в логе;
- полученный `AudioBuffer` кодируется в PCM16 WAV хелпером `audioBufferToWav` (`frontend/src/player/wav.js`,
  ~40 строк, без зависимостей) → `Blob({ type: "audio/wav" })`;
- не зависит от play/stop и не трогает живой `Tone.Transport`/Destination; можно звать во время воспроизведения.
- `song` опционален: без аргумента рендерится последний загруженный Song.

## 9. UX и анимации (по дизайн-системе)

Токены движения: `fast 120ms` (hover/иконки), `base 180ms` (overlay/тосты/тоглы), `pulse 700ms` (индикация
процесса), `beatFlash 160ms` (успех-флеш). Семантика акцентов: **cyan** = ввод/активный drop-таргет,
**lime** = валидно/успех, **coral** = ошибка, neutral raised = обычные действия. Правило дизайн-системы:
фиксируем размеры, анимируем **цвет/opacity/transform**, layout при фидбеке не прыгает.

**Transport-кнопки:** «Импорт», «Экспорт JSON», «Сохранить WAV» рядом с «Пример/Play/Stop» (стиль
`secondary-button`; иконки по возможности). Hover — `fast`.

**Overlay импорта:** появляется по кнопке «Импорт» или по `dragenter` файла над окном. Оверлей — fade
(`base`), диалог — лёгкий scale/slide-in (`base`, `ease-out`). Внутри:
- крупная **drop-zone** (dashed border, radius `--r-lg`): idle → нейтральная; `dragover` → cyan border +
  подсветка фона + подпись «Отпусти, чтобы импортировать» (переход `fast`);
- **textarea** для вставки JSON (моно-шрифт) + кнопка «Импортировать текст»;
- кнопка «Выбрать файл» (скрытый input).

**Состояния конвейера (внутри диалога, без скачков layout):**
- `parsing` — subtle shimmer/`pulse` по drop-zone;
- `success` — `beatFlash` lime, затем overlay закрывается, трек грузится, toast «Импортировано»;
- `error` — coral border + короткий shake (transform, один цикл), текст ошибки под зоной; overlay остаётся.

**WAV-рендер:** кнопка «Сохранить WAV» → состояние «Рендер…» с `pulse`-индикацией; по завершении — скачивание +
toast «WAV сохранён». Кнопки экспорта заблокированы во время рендера.

**Доступность:** overlay закрывается по `Esc` и клику по фону; фокус уводится в диалог и возвращается; drop-zone
и кнопки — с `aria-label`. При `prefers-reduced-motion: reduce` — отключаем shake/shimmer/scale, оставляем мгновенную
смену состояния и цвет.

## 10. Обработка ошибок

- Импорт: битый JSON → «Не удалось разобрать JSON». Инвариант-ошибки → первая-вторая проблемы человекочитаемо
  (напр. «bpm 400 вне диапазона 40..220», «пара sampler↔saw_lead недопустима»). Всё внутри диалога, состояние
  приложения не меняется. Дропнутые события → warning-toast, но трек грузится.
- WAV: любые исключения рендера → toast «Не удалось сохранить аудио», кнопки разблокируются.
- Всё не-фатально, консистентно с текущим toast/`message`-UX. Нет `song` → кнопки экспорта/WAV disabled.

## 11. Тестирование

- **Юнит (node --test, song-io чистый, без DOM):**
  - `serializeSong`→`parseSong` round-trip на `sample-song.json` без потерь;
  - отклоняются: `version !== 1`, `bpm` вне диапазона, пара `sampler↔synth-sound`, дубли `id`, битый JSON;
  - события со `step` вне лупа дропаются; `normalizeImportedSong` идемпотентен.
- **WAV/офлайн-рендер** — headless не тестируется (WebAudio); сценарий в
  `frontend/src/player/MANUAL_SMOKE_CHECK.md`: рендер → WAV скачивается, играется, длительность ≈ `bars*4*60/bpm` с.
- **Импорт-UI** — в manual smoke: три способа (textarea, файл, drag файла, drag текста) грузят один и тот же трек;
  битый ввод показывает ошибку и не роняет UI; проверка `prefers-reduced-motion`.
